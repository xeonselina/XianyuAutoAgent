## 背景
顺丰时效预估已经存在于 `ai_kefu/tools/calculate_logistics.py`，但库存管理系统拥有独立的运行环境和前端应用。PC 端和移动端都支持新建及编辑租赁记录，因此警告逻辑需要使用统一的预估来源，同时保留两端现有的提交流程。

## 目标与非目标
- 目标：保证库存管理系统各客户端使用一致的顺丰预估结果；在数据保存前警告；允许用户明确确认后继续。
- 非目标：自动修改物流天数、禁止保存物流时间较短的档期、调用顺丰实时路由规划 API，或改变现有租赁数据结构。

## 技术决策
- 在库存管理系统内增加一个轻依赖的预估工具，映射现有的深圳至目的地顺丰标快规则。这样可以避免库存管理系统启动时依赖 AI 客服包及其传递依赖。
- 通过现有租赁 API 提供预估能力，使 PC 端和移动端共用同一套规则实现。
- 目的地为空时不参与比较，保留当前收货地址可选填的行为。
- 仅当“所选物流天数小于预估物流天数”时警告；相等或更长时正常提交。
- 使用各自 UI 框架的警告确认框：PC 端使用 `Element Plus`，移动端使用 `Vant`，均提供明确的继续和取消操作。

## 风险与权衡
- 预估工具映射的原始规则仍位于 AI 客服包中，未来存在规则不一致的风险。测试将固定代表性地区的结果，库存管理工具中也会注明规则来源。
- 提交时会增加一次同源预估请求。该操作只执行本地确定性计算，延迟很低。

## 发布方案
后端接口应先于或连同重新构建的 PC 端、移动端静态资源一起发布。不需要进行数据迁移。

## 待确认问题
- 无。警告按需求保持为提示性质，不强制阻止保存。
