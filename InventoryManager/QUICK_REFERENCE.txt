KEY COMPLEXITY DRIVERS FOR MOBILE:
──────────────────────────────────

1. DATE INTERDEPENDENCY
   • startDate changes → clears endDate if invalid
   • endDate changes → triggers availability check
   • Both changes → resets entire availability state
   • minSelectableDate computed from rental.start_date

2. WATCH AUTO-FILLING
   • destination field → auto-extracts phone number
   • ONLY fills customerPhone if currently empty
   • Uses regex pattern: /^1[3-9]\d{9}$/
   • Shows success message after extraction

3. ASYNC DATA FETCHING
   • xianyuOrderNo button → POST /api/rentals/fetch-xianyu-order
   • Auto-fills: destination, buyerId, orderAmount, customerPhone
   • Converts: pay_amount (cents) → yuan with .toFixed(2)
   • Destination format: "name phone prov+city+area+town+address"

4. AVAILABILITY CHECKING
   • Triggered on device selector focus
   • Triggered on accessory selector focus
   • Checks enabled only AFTER dates selected
   • Shows "档期不可用" tags on conflicted items
   • Disables select options if unavailable & checked

5. DEVICE/ACCESSORY FILTERING
   • Devices: filtered by name including serial_number display
   • Phone holders: filtered by name/model matching "手机支架" or "phone"
   • Tripods: filtered by name/model matching "三脚架" or "tripod"
   • All filters use .includes() + .toLowerCase() matching

6. ACCESSORY SUMMARY (Conditional Display)
   • Section visible ONLY if: hasSelectedAccessories = true
   • hasSelectedAccessories = bundledAccessories.length > 0 
                            || phoneHolderId !== null 
                            || tripodId !== null
   • Shows bundled & inventory items with remove buttons
   • Bundled items: labeled "(配套附件)"
   • Inventory items: labeled "(库存)" with delete buttons

7. FORM TRANSFORMATION
   CREATE payload:
   {
     device_id, start_date, end_date, customer_name, customer_phone,
     destination, ship_out_time, ship_in_time,
     includes_handle (boolean), includes_lens_mount (boolean),
     accessories (array of IDs), xianyu_order_no, order_amount,
     buyer_id, photo_transfer
   }
   
   EDIT payload:
   Same as CREATE but NO start_date, customerName field
   (Also includes tracking numbers & status)

8. VALIDATION RULES
   CREATE requires:
   ✓ startDate, endDate, logisticsDays, customerName
   ✗ customerPhone (optional, regex if provided)
   ✗ destination (optional)
   
   EDIT requires:
   ✓ deviceId, endDate
   ✗ customerPhone (optional, regex if provided)
   ✗ destination (optional)

9. CONFLICT DETECTION
   Device conflict:
   • Checked when device selector changes
   • Shows MessageBox warning if conflict detected
   • Resets to previous deviceId if user cancels
   
   Accessory conflict:
   • Checked when accessory selector changes
   • Shows MessageBox warning if conflict detected
   • Removes accessory from form if user cancels
   
   Duplicate check:
   • Checked before CREATE submit
   • Compares customerName + destination
   • Shows all duplicates if found
   • Requires user confirmation to proceed

API ENDPOINTS CALLED:
────────────────────

1. POST /api/rentals/fetch-xianyu-order
   Request: { order_no: string }
   Response: { success, data: {...}, message }
   Auto-fills: destination, buyerId, orderAmount, customerPhone

2. ganttStore.createRental(data)
   Creates new rental record

3. ganttStore.updateRental(id, data)
   Updates existing rental record

4. ganttStore.findAvailableSlot(startDate, endDate, logisticsDays, modelId, isAccessory)
   Finds available device/accessory for dates

5. conflictDetection.checkDeviceConflict({deviceId, startDate, endDate, excludeRentalId})
   Checks if device has conflicts

6. conflictDetection.checkDuplicateRental({customerName, destination})
   Checks for duplicate bookings

7. availability.checkDevicesAvailability(devices, params)
   Checks all devices availability

8. availability.checkAccessoriesAvailability(accessories, params)
   Checks all accessories availability
