# 闲鱼漏录订单告警设计

## 背景

库存管理系统通过 `rentals.xianyu_order_no` 关联闲鱼订单。闲鱼上已经付款、等待发货的订单如果没有在库存管理中预定，就不会进入甘特图、面单打印和发货流程，存在漏发货风险。

闲管家订单列表接口能够返回订单号、状态、实付金额、买家昵称、收件手机号、地址和商品信息。系统已有闲管家签名客户端、订单详情查询能力、现有预定弹框 `BookingDialog` 和 APScheduler。

## 目标

- 每 10 分钟检查单一已绑定闲鱼店铺的待发货订单。
- 找出 `order_status = 12`、`pay_amount > 5000` 分且没有出现在 `rentals.xianyu_order_no` 中的订单。
- 仅在甘特图顶部显示告警条，不新增页面、抽屉或独立预定界面。
- 从告警条打开并复用现有 `BookingDialog` 完成补录。
- 允许用户填写原因后永久标记订单为“无需录入”，不提供恢复入口。
- 外部接口失败时保留上一次成功结果，避免将未知状态错误显示为“没有漏单”。

## 非目标

- 不自动创建预定。
- 不推断开始/结束时间、设备型号、具体设备或附件。
- 不处理退款状态。
- 不接入闲管家订单推送回调。
- 不支持多店铺选择。
- 不新增忽略记录恢复界面。

## 架构

### 闲管家客户端

在现有 `XianyuOrderService` 中增加订单列表查询：

- 调用 `POST /api/open/order/list`。
- 固定查询 `order_status = 12`。
- 每页 100 条，依据接口返回的 `count` 完成全部分页。
- 任意一页失败时整次查询失败，不返回部分结果。
- 复用现有请求体签名、超时和凭证配置。

### 对账服务

新增 `XianyuOrderReconciliationService`，负责：

1. 拉取完整的待发货订单列表。
2. 保留 `pay_amount > 5000` 分的订单。
3. 排除已经出现在任意 `Rental.xianyu_order_no` 中的订单。
4. 排除已永久忽略的订单。
5. 在单一数据库事务中更新告警缓存和同步状态。
6. 只有完整查询成功后才删除已不再符合条件的待处理告警。

订单号作为所有比较和去重的唯一业务键，并统一转为去除首尾空格的字符串。

### 调度与并发

- 复用现有 APScheduler，每 10 分钟执行一次对账。
- 甘特图进入时先读取缓存，再请求一次立即刷新。
- 用户可以从告警条手动刷新。
- 定时、页面进入和手动刷新共享互斥锁，保证同一时刻最多有一个闲管家列表查询。
- 如果已有刷新正在运行，重复刷新返回当前缓存及“正在刷新”状态，不重复访问外部接口。

## 数据模型

### `xianyu_order_alerts`

保存当前待处理告警和永久忽略记录：

- `id`
- `order_no`：唯一索引
- `state`：`pending` 或 `ignored`
- `pay_amount`：整数，单位为分
- `buyer_nick`
- `receiver_name`
- `receiver_mobile`
- `address`
- `goods_title`
- `goods_sku_text`
- `order_time`
- `first_detected_at`
- `last_seen_at`
- `ignored_reason`
- `ignored_at`
- `created_at`
- `updated_at`

`pending` 记录在订单已录入、已不再待发货或金额不再满足条件时清理。`ignored` 记录永久保留，即使订单暂时不再待发货，之后重新进入待发货状态也不会再次告警。

### `xianyu_order_sync_state`

使用单例记录保存：

- `last_attempt_at`
- `last_success_at`
- `last_error`

它确保即使当前没有告警，前端仍能判断检查是否成功以及数据是否最新。

## 内部 API

### `GET /api/xianyu-order-alerts`

返回：

- 当前仍未录入的 `pending` 告警。
- 告警总数。
- 上次尝试和成功时间。
- 最近错误。
- 是否正在刷新。

读取时再次排除已经存在于 `rentals` 的订单，因此预定成功后无需等待下一次外部同步即可消失。

### `POST /api/xianyu-order-alerts/refresh`

触发完整对账并返回最新缓存。重复调用不得并发访问闲管家。

### `POST /api/xianyu-order-alerts/<order_no>/ignore`

请求体包含非空 `reason`。操作前验证订单存在；成功后将记录永久改为 `ignored`。前端必须二次确认“无法恢复”。

## 甘特图交互

警告条位于甘特图工具栏下方、过滤器上方。

- 无告警且最近检查成功时不显示。
- 有告警时显示红色条幅、数量、“展开详情”和“刷新”。
- 刷新中显示轻量加载状态，不阻塞甘特图。
- 刷新失败时保留旧告警并展示上次成功时间和失败提示。

展开内容仍属于警告条区域，以内联列表展示：

- 买家昵称
- 手机号
- 实付金额
- 下单时间
- 商品名称和规格
- 闲鱼订单号
- “去补录”
- “无需录入”

“去补录”将订单号传给现有 `BookingDialog`。弹框打开后调用现有订单详情拉取逻辑，自动填入订单号、昵称、手机号、地址和金额。用户按原流程填写日期、设备、型号和附件。创建成功后重新读取告警列表。

“无需录入”要求填写原因并二次确认永久忽略，成功后立即从告警条移除。

## 失败处理

- 闲管家超时、鉴权失败、业务错误或分页不完整时，不修改现有告警集合。
- 数据库更新失败时回滚整次对账。
- 日志不得记录应用密钥、完整签名 URL 或买家手机号。
- 忽略与补录并发时，已录入订单始终不显示。
- 首次检查失败且没有成功缓存时，甘特图显示“暂时无法检查漏录订单”，不得显示“没有漏单”。

## 测试与验收

- `order_status = 12` 且 `pay_amount = 5001`：未录入时显示。
- `pay_amount = 5000`：不显示。
- 非待发货状态：不显示。
- 已存在相同 `xianyu_order_no`：不显示。
- 任何退款状态均不参与过滤。
- 永久忽略后不再显示。
- 多页订单完整拉取后才更新缓存。
- 任意分页失败时保留旧缓存。
- 预定弹框被复用并正确拉取订单信息。
- 补录成功后告警立即消失。
- 定时、进入页面和手动刷新并发时只进行一次外部查询。
