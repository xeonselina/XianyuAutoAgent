# 甘特图一键重排档期设计

## 背景

库存管理甘特图按设备展示 rental 的物流占用区间。运营人员希望在不改变订单日期的前提下，把今天及以后的未寄出主 rental 重新分配到同型号设备，减少设备档期之间的空隙。同时，系统中存在主 rental 与附件子 rental、已经预约或寄出的固定订单，以及人工安排的客户间直接转寄（接力）场景，重排必须保护这些关系且不能丢失任何 rental。

本变更还修复“新建 rental → 查找档期”遗漏设备生命周期状态过滤的问题。当前查询只限制 `Device.status == 'online'`，没有同时限制 `Device.lifecycle_status == 'active'`。

## 目标

- 在 PC 甘特图提供全型号一键重排，并强制先预览后执行。
- 使用 OR-Tools CP-SAT 按 `model_id` 独立优化主设备分配。
- 只修改符合条件的主 rental 的 `device_id`；所有日期、状态和订单字段保持不变。
- 保持全部子 rental 的附件设备及父子关系不变。
- 支持人工确认并永久保存接力绑定。
- 普通档期最多允许前单收回日与后单寄出日为同一自然日。
- 通过快照、行锁、单事务、完整性校验和审计日志确保失败时零部分更新。
- 新建 rental 查找档期与重排统一只使用在线且生命周期为使用中的设备。

## 非目标

- 不重排子 rental 或附件设备。
- 不修改 `start_date`、`end_date`、`ship_out_time` 或 `ship_in_time`。
- 不跨型号分配设备。
- 不自动修改已预约、已寄出或进行中的 rental。
- 不在测试中连接或写入生产数据库。
- 不承诺在三秒求解上限内证明全局最优；超时时允许展示带状态说明的合法可行解。

## 领域定义

### 可移动主 rental

必须同时满足：

- `parent_rental_id IS NULL`
- `status = 'not_shipped'`
- `ship_out_time IS NOT NULL`
- `DATE(ship_out_time) >=` 应用时区 `Asia/Shanghai` 的今天
- 关联主设备具有非空 `model_id`

`ship_out_time` 为空或 `model_id` 为空的记录不参与重排，在预览中列为跳过项并说明原因。

### 固定 rental

除 cancelled 外，会与优化时间范围相交但不满足可移动条件的主 rental 都作为固定占用锚点。特别包括：

- `scheduled_for_shipping`
- `shipped`
- 正在进行或已经寄出的 rental
- 寄出日期早于今天的 rental

固定 rental 的设备不可改变。Cancelled rental 不占用档期。

### 合法目标设备

目标设备必须同时满足：

- 与 rental 原主设备的 `model_id` 相同
- `is_accessory = false`
- `status = 'online'`
- `lifecycle_status = 'active'`

非使用中设备上的固定 rental 仍保留在原设备，但该设备不得接收新的重排档期。预览后目标设备状态发生变化会使快照失效。

### 普通档期重叠

普通档期以物流日期建立半开区间：

```text
[DATE(ship_out_time), DATE(ship_in_time))
```

因此，后一 rental 的寄出日期可以等于前一 rental 的收回日期，表示早上收货、下午重新寄出。若后一寄出日期早于前一收回日期，则属于超过允许范围的重叠，必须拆分到不同设备或建立人工接力绑定。

## 接力绑定

### 数据模型

新增 `rental_relay_bindings`：

| 字段 | 说明 |
| --- | --- |
| `id` | 主键 |
| `predecessor_rental_id` | 前一主 rental 外键 |
| `successor_rental_id` | 后一主 rental 外键 |
| `confirmed_at` | 人工确认时间 |
| `created_at` / `updated_at` | 时间戳 |

约束：

- `(predecessor_rental_id, successor_rental_id)` 唯一。
- 同一 predecessor 最多绑定一个 successor，同一 successor 最多绑定一个 predecessor，由此可形成 A → B → C 链但不能分叉。
- 前后记录必须是主 rental、同一 `model_id`、确认时位于同一设备，且顺序与物流日期一致。
- 删除 rental 时级联删除绑定元数据，不改变现有 rental 删除语义。

### 识别与交互

点击“一键重排档期”后，后端先扫描同设备上按寄出时间排序的相邻主 rental：

- 已保存绑定：默认保持，界面标记“已绑定接力”，允许人工选择解除。
- 超过一天重叠但未绑定：必须逐项选择“保持接力并永久绑定”或“解除接力，分开重排”。
- 未处理完全部待确认项时，不生成最终优化预览。
- 两笔均为固定 rental 且已重叠时无法拆分，只能保持接力或取消该型号重排。

选择保持后，相关 rental 在求解中成为不可拆分的接力档期块。数据库记录仍保持独立，块内允许已确认重叠，且所有成员强制分配到同一设备。接力链只要包含一个固定 rental，整个链固定在当前设备。

新建或解除绑定与设备重排在最终执行的同一事务中落库；取消预览不会写入绑定。

## 两步用户流程

### 第一步：确认接力关系

弹窗展示每组重叠双方的：

- 型号和当前设备
- 姓名
- 电话号码
- 地址
- 寄出与收回日期
- 重叠天数
- 当前是否已有永久绑定

用户完成全部选择后点击“确认接力选择并计算预览”。

### 第二步：预览并执行

预览按型号汇总：

- 调整的主 rental 数量
- 优化前后承载可移动档期的设备数量
- 接力绑定数量
- 跳过、无解和异常数量
- 每笔记录的姓名、电话、地址、档期、原设备和目标设备

未变化记录默认折叠，接力记录明确标识。页面提供“返回修改接力”和“执行重排”。不增加第三步确认页；“执行重排”按钮即提交动作。

## API 与数据流

### `POST /api/gantt/reorder/analyze`

只读扫描现有接力绑定和未确认重叠，返回第一步数据。无数据库写入。

### `POST /api/gantt/reorder/preview`

请求包含第一步的保持/解除选择。服务端校验选择后按型号运行 OR-Tools，返回第二步明细及十分钟有效的签名预览令牌。无数据库写入。

令牌由服务端签名，包含：

- 唯一 nonce 和过期时间
- 相关数据快照哈希
- 用户的接力选择
- 服务端生成的完整目标分配
- 求解状态和配置版本

客户端不能提供或修改目标设备映射。

### `POST /api/gantt/reorder/execute`

请求只提交签名预览令牌。后端：

1. 验证签名和十分钟有效期。
2. 使用 `SELECT ... FOR UPDATE` 锁定相关主 rental、全部子 rental、设备和绑定。
3. 重建快照并比较哈希。
4. 使用令牌内的目标映射建立固定分配的 CP-SAT 模型，重新验证可行性，而不是重新选择不同结果。
5. 执行完整性校验。
6. 在一个事务中更新绑定、可移动主 rental 的 `device_id` 及审计日志。
7. 提交前再次刷新并校验全部不变量。

成功响应返回与预览一致的型号汇总和实际变更。成功后前端刷新甘特图。

## OR-Tools 模型

### 预处理

- 每个 `model_id` 独立建立模型，禁止跨型号变量。
- 接力链折叠成一个逻辑档期块，块区间为所有成员物流日期区间的并集。
- 未绑定主 rental 各自形成档期块。
- 固定档期块只允许当前设备；可移动档期块允许所有合法同型号目标设备。

### 硬约束

- 每个可移动档期块恰好分配到一台候选设备。
- 普通档期块在同一设备上使用 `NoOverlap`，允许半开区间首尾同日衔接。
- 接力链成员分配变量相等。
- 固定档期块的设备变量固定。
- 所有目标设备必须是同型号、非附件、online、active。
- 每个可移动主 rental 在输入和输出中恰好出现一次。

### 分层优化目标

按优先级顺序求解并固定上一层最佳已知值：

1. 最少使用承载可移动未来档期的目标设备。
2. 在设备数量相同下，最小化相邻逻辑档期块之间的总空闲天数。
3. 在压缩效果相同下，最少修改主 rental 的 `device_id`。

每个型号总求解时间最多三秒，`num_search_workers = 1` 并使用固定随机种子，以提高预览复现性并控制 NAS CPU。求解状态为：

- `OPTIMAL`：展示最优结果。
- `FEASIBLE`：展示“在时间限制内找到合法方案，未证明最优”。
- `INFEASIBLE` / `UNKNOWN`：该型号保持原分配并展示原因。

有合法结果的型号可执行；无解型号保持原状。执行阶段任一快照或完整性校验失败时，整次操作全部回滚。

## 完整性与并发安全

快照覆盖：

- 所有相关主 rental 的 ID、设备、型号、日期、物流时间、状态和 `updated_at`
- 每个主 rental 的完整子 rental ID 集合
- 子 rental 的设备、父 ID、日期、物流时间、状态和 `updated_at`
- 候选设备的型号、设备状态、生命周期状态和 `updated_at`
- 现有接力绑定及本次用户选择

事务提交前后必须满足：

- 主 rental 总数和 ID 集合完全一致。
- 子 rental 总数和 ID 集合完全一致。
- 只允许可移动主 rental 的 `device_id` 改变。
- 子 rental 的 `device_id`、`parent_rental_id`、日期、物流时间和状态不变。
- 主 rental 的日期、物流时间、状态、型号和子 rental 集合不变。
- 每个可移动主 rental 恰好映射一次。
- 普通档期不存在超过同日衔接的重叠。
- 接力链保持在同一设备。
- 所有目标设备仍为 online 且 active。

审计日志与设备更新使用同一事务写入，记录操作批次、rental ID、原设备、目标设备和接力变更。任何异常调用 `rollback()`；禁止循环逐单提交。

重复提交旧令牌时，首次执行造成的快照变化会使后续请求失败；无变化预览不会产生重复审计副作用。

## 查找档期生命周期状态 Bug

根因是 `GanttService.find_available_slot()` 的两条设备查询路径仅过滤 `Device.status == 'online'`，遗漏 `Device.lifecycle_status == 'active'`。修复范围覆盖主设备和附件查找：

```text
Device.status == 'online'
AND Device.lifecycle_status == 'active'
```

重排候选设备复用相同的“在服务中”查询规则，防止两个功能再次出现资格判断分叉。`InventoryService.check_device_availability()` 仍负责档期冲突检测，设备资格在候选查询入口统一保证。

## Docker 依赖

生产依赖固定 `ortools==9.15.6755`。项目现有 Python 3.10、Debian 12 slim 和 buildx 多架构流程支持 Linux amd64/arm64 预编译 wheel。实测：

- OR-Tools wheel 约 28–30 MB。
- 包含 NumPy、Pandas 等依赖后下载约 54 MB。
- Python site-packages 安装后约增加 232 MB。

OR-Tools 随应用镜像安装，NAS 主机无需单独安装。构建必须验证 amd64 和 arm64 均可导入 `ortools.sat.python.cp_model`。

## 异常处理

- 无可移动 rental：提示当前没有需要重排的档期，不写数据库。
- 未确认重叠：阻止预览并定位到待处理项。
- 无法拆分的固定重叠：要求保持接力或取消该型号。
- 绑定跨型号、指向子 rental、顺序异常或不在同一设备：阻止该型号并提示人工检查。
- 令牌无效或过期：要求重新预览。
- 快照、子 rental、设备状态或绑定变化：返回冲突响应并要求重新预览。
- 某型号无解：该型号保持原分配，其他合法型号仍可预览。
- 执行时任一错误：整次事务回滚，前端保留错误信息并刷新当前甘特图数据。

## 测试数据库隔离

自动化测试绝不加载生产 `.env`，绝不连接或写入 `192.*` 生产数据库。

默认策略：

- 纯优化逻辑使用合成 fixture 和无数据库单元测试。
- 普通服务测试可使用内存数据库。
- MySQL 行锁、事务与回滚测试使用独立 `inventory_mysql_test` 容器、独立网络、临时 volume 和包含 `test` 的数据库名。
- 测试只接受显式 `TEST_DATABASE_URL` 和 `TESTING=true`。
- 启动保护发现数据库主机为 `192.*`，或数据库名不含 `test`，立即终止。

仅在合成数据不能复现生产问题、并获得单独授权后，才允许使用只读账号执行 `mysqldump --single-transaction`。Dump 必须导入隔离容器并脱敏姓名、电话、地址、订单号和快递单号；dump 与测试 volume 不进入 Git。

## 测试与验收

### 后端

- 同型号可重排，跨型号无分配变量。
- 只移动今天及以后寄出的 not_shipped 主 rental。
- scheduled_for_shipping、已寄出和进行中 rental 固定。
- online + active 才能成为目标设备。
- 查找档期排除 offline、sold、damaged、decommissioned、retired。
- 普通档期允许同日衔接，禁止更长重叠。
- 未确认重叠阻止预览。
- 已确认接力和 A → B → C 链保持同设备。
- 接力链包含固定 rental 时整链固定。
- 父 rental 换设备后全部子 rental 和附件设备不变。
- 预览零写入。
- 过期令牌、快照变化和设备状态变化拒绝执行。
- 注入中途异常后全部设备分配、绑定和审计日志回滚。
- 执行前后主、子 rental ID 集合和数量一致。
- 无解和超时无合法解时原分配不变。
- 重复令牌不产生重复副作用。

### 前端

- 两步流程完整展示姓名、电话、地址、档期和设备变化。
- 未确认接力时禁用预览。
- 预览可返回修改接力。
- 第二步直接执行，不出现第三步确认页。
- 快照冲突和过期提示引导重新预览。

### 部署

- amd64 和 arm64 Docker 构建成功。
- 两种镜像内均可导入 CP-SAT。
- 运行重排时遵守每型号三秒、单工作线程限制。

最终验收要求：预览与实际分配完全一致；除允许的主 rental `device_id` 外零字段变化；固定 rental、子 rental 和订单日期零变化；所有普通重叠及接力绑定合法；任何失败场景下数据库保持执行前状态。
