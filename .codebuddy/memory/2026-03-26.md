## 评估回放系统 (Eval Replay)

创建了 prompt/模型 A/B 测试的批量回放评估工具，包括两个脚本和一张 MySQL 表：

### 新建文件
- `ai_kefu/scripts/eval_replay.py` — 批量回放脚本：从 conversations 表取用户消息+上下文，用当前 Agent 系统重新生成 AI 回复，结果写入 `eval_runs` 表
- `ai_kefu/scripts/eval_analyze.py` — 分析脚本：对比 AI 回复 vs 人类回复（Jaccard 相似度 + 长度比），生成统计报告、提取高/低相似度样例，支持两次评估对比、导出 JSON/Markdown

### 新建 MySQL 表
- `eval_runs` — 存储评估结果，包含原始消息、人类回复、AI 回复、上下文、工具调用、耗时、模型名等

### Makefile 新命令
- `make eval-replay ARGS="--limit 50 --tag v3"` — 执行回放
- `make eval-analyze ARGS="--run-id xxx"` — 分析结果
- `make eval-list` — 列出历史评估运行
- 支持 `--model` 覆盖模型、`--dry-run` 预览、`--compare` 对比两次评估

## eval_replay.py 自动过滤卖家消息 (规则固化)

将卖家过滤规则从手动传参改为**自动执行**，避免每次评估时忘记排除：

### 新增 `fetch_seller_user_ids()` 函数
- 自动从 `conversations` 表查出所有 `message_type='seller'` 的 `DISTINCT user_id`
- 这些 user_id 即使有些消息被标记为 `message_type='user'`，实际仍是卖家身份
- SQL: `SELECT DISTINCT user_id FROM conversations WHERE message_type = 'seller'`

### 改造 `fetch_eval_candidates()`
- 参数从 `exclude_user_id: str` 改为 `exclude_user_ids: List[str]`（支持多个）
- SQL 从 `c.user_id != %s` 改为 `c.user_id NOT IN (%s, %s, ...)` 参数化查询

### 改造 `main()`
- 在提取候选之前自动调用 `fetch_seller_user_ids()` 获取卖家列表
- `--exclude-user-id` 参数保留但改为 `nargs="*"` 支持多个值，作为手动补充
- 运行时自动打印排除的 user_id 列表，方便确认

### 自动过滤规则
| 过滤条件 | 实现方式 |
|---|---|
| `message_type='seller'` 的消息 | `c.message_type = 'user'` 条件天然排除 |
| 卖家 user_id 发的所有消息 | `fetch_seller_user_ids()` 自动查询 + `NOT IN` 排除 |
| 额外手动排除 | `--exclude-user-id` 参数补充 |

## parse_date 日期推断规则增强

增强 `tools/parse_date.py` 中的 `_resolve_date()` 函数，固化日期推断规则：

- 新增 `only_day` 参数，区分"只说日号"（如"20号"）和"带月份"（如"3月20号"）
- **只说日号且日号 < 今天** → 推断为下个月（`only_day=True`）
- **当月+日号已过** → 推断为下个月（如当月说"3月15号"但15号已过）
- **非当月+已过** → 推到明年（如3月说"1月15号"）
- 在"X号"和"X号收/寄回"等只有日号的匹配分支中传入 `only_day=True`
- 添加 `logger.info` 详细记录日期推断过程，方便调试
- 更新工具定义的 description，加入日期推断规则说明

## 优化 AI 调用成功率 (Agent API 500 错误)

修复了大量 `[Agent API Error] Server error '500 Internal Server Error'` 问题，改动4个文件：

### 1. `xianyu_interceptor/http_client.py` — 扩展重试范围
- 原来只重试 `ConnectError` 和 `PoolTimeout`，**500/502/503/504 和 ReadTimeout 不重试**
- 新增 `_is_retryable_error()` 判断函数，将服务端5xx错误纳入重试
- 重试次数从3次增加到4次，退避上限从10s提高到15s
- 添加 tenacity `before_sleep_log` 记录每次重试日志

### 2. `api/routes/chat.py` — 不再笼统抛500
- 原来 `executor.run()` 返回 `{"error": ...}` 时直接抛 `HTTPException(500)`
- 改为返回正常 HTTP 200 响应，将 error 放在 `metadata.error` 字段
- 这样 http_client 可以区分：真正的服务端崩溃(500) vs 业务级LLM错误(200+error status)

### 3. `agent/executor.py` — 临时错误优雅降级
- 对 Qwen API 常见临时错误（RequestFailure, ServiceUnavailable, Throttling, timeout等），自动生成友好 fallback 回复 "抱歉，系统暂时繁忙，请稍后再试~"
- 避免返回空 response 导致用户收不到任何回复

### 4. `xianyu_interceptor/message_handler.py` — 处理 error status 响应
- 新增对 `agent_response.metadata.error` 的检查
- 如果 Agent 返回了 fallback response，直接使用
- 如果没有 fallback，使用配置的 `fallback_message`

## 修复超时链路问题 (API 500 根因)

发现 API 500 错误的核心根因是**超时链路不匹配**：

### 问题诊断
- DashScope SDK `Generation.call()` 默认 `request_timeout=300s`（5分钟！）
- 重试策略：5 次重试，指数退避 2-60s
- **最坏情况**：300×5 + 退避 = 1500+s
- 但 interceptor httpx 只等 120s → **必然超时报 500**
- `executor.py` 虽然定义了 `turn_timeout_seconds=120` 但**从未强制执行**

### 修复内容（4个文件）

#### `config/constants.py`
- `QWEN_API_RETRY_ATTEMPTS`: 5 → **3**
- `QWEN_API_RETRY_DELAY`: 2.0 → **1.0** 秒
- 新增 `QWEN_API_RETRY_MAX_DELAY`: **10.0** 秒
- 新增 `QWEN_API_TIMEOUT`: **30.0** 秒

#### `config/settings.py`
- 新增 `qwen_api_timeout: float = 30.0`（可通过 .env 配置）
- `turn_timeout_seconds`: 120 → **100**

#### `llm/qwen_client.py`
- `call_qwen()` 和 `stream_qwen()` 都加了 `request_timeout=30s`
- 重试延迟上限从 60s 降为 10s
- 最坏总耗时：30+1+30+3+30 ≈ **94s** < interceptor 120s ✅

#### `agent/executor.py`
- 导入 `TurnTimeoutError`
- 在 `run()` 中用 `threading.Timer` + `threading.Event` 实现超时保护
- 超过 `turn_timeout_seconds`(100s) 后设置超时标志，下次循环检查时抛出 `TurnTimeoutError`
- 新增 `TurnTimeoutError` 独立 except 分支，设置 `TerminateReason.TIMEOUT`
- 添加 `finally` 块确保定时器被取消

### 优化后的超时链路
```
Qwen API 单次: 30s  →  重试3次(+退避1-10s)  →  最坏~94s
Agent executor: 100s 超时保护
interceptor httpx: 120s 总超时（含连接10s）
```

## LLM 客户端迁移：DashScope SDK → OpenAI SDK

### 背景
- `.env` 中 `MODEL_NAME=qwen3.5-plus` 调用报错 `RetryError[RequestFailure]`

### 迁移方案
将 `llm/qwen_client.py` 从 DashScope SDK 改为 OpenAI SDK，**保持返回值格式完全一致**（dict: `{"choices": [{"message": {...}}]}`），下游代码零改动。

## 修复 eval_replay.py 过滤规则和 human_reply 逻辑

### 问题 1：human_reply 匹配了【调试】消息
`eval_replay.py` 的 `human_reply` 子查询只按 `message_type = 'seller'` 找下一条卖家消息，但没有排除【调试】前缀的 AI 回复。

### 问题 2：过度排除导致 0 条结果
加了 `user_id='2200687521877'` + `NOT LIKE '【调试】%%'` 后，数据库里 755 条卖家 seller 消息**全部**是【调试】开头或 API Error，真正的人工回复为 **0 条**，所以 HAVING human_reply IS NOT NULL 过滤后结果为空。

### 问题 3：输入消息也不对
原过滤规则 `user_id='2200687521877' OR message_type='seller'` 把卖家自己的调试回复也当作了 eval 输入。

### 最终修复（用户确认人工回复定义）
用户明确人工回复定义：`(user_id='2200687521877' OR message_type='seller') AND content NOT LIKE '【调试】%'`

数据验证：
- 759 条 `message_type='user'` + `user_id='2200687521877'` = 卖家手动在 App 发的消息（真人工回复）
- 776 条 `message_type='seller'` + 非调试 = 全是 API Error（排除）

修复 `eval_replay.py`：
1. **输入消息**: `message_type='user' AND user_id != '2200687521877'`（买家消息）
2. **human_reply 子查询**: `(s.user_id = '2200687521877' OR s.message_type = 'seller') AND NOT LIKE '【调试】%' AND NOT LIKE '%Agent API Error%' AND NOT LIKE '%Unexpected Error%'`
3. dry-run 验证通过，找到有效的买家消息→人工回复配对

## 修复 eval_replay session 创建慢的问题

### 问题
eval 运行时出现 "Session not found in store, creating new session" 日志后等待很久。

### 根因
`eval_replay.py` 之前只在 `context_summary` 非空时才预写入 Redis session。当某条消息前面没有历史对话时，session 不会预写入 Redis → `executor._get_or_create_session()` 找不到 → 走 `_load_history_as_context()` 慢路径（查 MySQL 30 条 + 对话 > 10 条时调 Qwen LLM 做摘要，最坏 ~94 秒）。

### 修复
1. **无论 context_summary 是否为空都预写入 Redis session**（第 273-287 行）
2. 空 context 时设置占位符 `"(无历史对话)"`，防止 executor 第 435 行条件判断再次触发慢路径

## eval_replay.py 并行化改造

### 背景
串行执行 eval 太慢，每条回放要调 LLM API（几秒到几十秒），50 条需要几十分钟。

### 改动
1. **新增 `--concurrency` / `-j` 参数**：控制并行工作线程数，默认 1（串行），建议 3-8
2. **新增 `_replay_worker()` 函数**：单条回放的 worker，每个 worker 创建独立的 DB 连接
3. **并行模式**：`-j > 1` 时使用 `ThreadPoolExecutor` 并发执行回放
4. **串行模式向后兼容**：`-j 1` 时直接循环调用 worker，不启动线程池
5. **线程安全**：
   - 消除 `settings.model_name` 的共享写入竞态（改为只读 `effective_model` 本地变量）
   - 每个 worker 独立 DB 连接（`conn_factory`）
   - 计数器用 `threading.Lock` 保护
6. **完成后显示总耗时和平均耗时**

### 用法
```bash
# 5 线程并行回放 100 条
python -m ai_kefu.scripts.eval_replay --limit 100 -j 5
```

## analyze_llm_quality.py v3 重写 — 基于 eval_runs 表

### 背景
旧版 analyze_llm_quality.py 从 conversations 表直接分析历史消息，逻辑复杂且与 eval_replay 流程脱节。

### 改动
完全重写 `ai_kefu/scripts/analyze_llm_quality.py`，改为从 `eval_runs` 表读取数据：

1. **数据来源**：`eval_runs` 表（eval_replay 产生的 user_message + human_reply + ai_reply 三元组）
2. **LLM 逐条评分**：用 Qwen 对每条数据做 5 维评分（简洁度/自然度/准确性/理解力/转化力 + 综合）
3. **并行评分**：默认 10 线程 `ThreadPoolExecutor` 并行调用 LLM
4. **Markdown 报告**：包含总体统计、评分汇总、分数分布、关键差异、AI 问题、Prompt 改进建议、最低/最高案例、逐条明细表
5. **终端报告**：简要汇总 + 最低/最高 5 条
6. **CLI 参数**：`--run-id`, `--list-runs`, `-j`, `--no-llm-eval`, `--output`, `--no-terminal`

### Makefile
新增 `make eval-quality ARGS="..."` 命令

### 用法
```bash
# 分析最近一次 eval run（10 线程并行 LLM 评分）
python -m ai_kefu.scripts.analyze_llm_quality

# 指定 run_id
python -m ai_kefu.scripts.analyze_llm_quality --run-id "eval_xxx"

# 跳过 LLM 评分，仅统计
python -m ai_kefu.scripts.analyze_llm_quality --no-llm-eval
```
