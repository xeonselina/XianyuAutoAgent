## ai_kefu: 修复新 session 缺少对话历史上下文的问题

**问题**：当 Redis session 过期（TTL 30分钟）或服务重启后，`session_mapper` 生成新的 `agent_session_id`，`_get_or_create_session` 在 Redis 中找不到，于是创建空 Session（`messages=[]`）。LLM 收到的 messages 只有 system prompt + 当前用户消息，完全没有之前的对话历史。

**根因数据验证**：
- 数据库中 session `f9d4356d` 的 turn 1 的 `llm_input` 只有 2 条 messages（system + user），同一 chat_id 之前有 14 轮对话全部丢失
- MySQL `conversations` 表持久化了所有历史消息，但 Agent 端从未读取过

**修复内容**：
1. **`api/routes/chat.py`**：sync 和 stream 路由都把 `request.context`（含 `conversation_id` 即 chat_id）传给 `executor.run()`/`executor.stream()`
2. **`agent/executor.py`**：
   - `run()` 和 `stream()` 新增 `context` 参数
   - `_get_or_create_session()` 新增 `chat_id` 参数
   - 新增 `_load_history_as_context(session, chat_id)` 方法：
     - 从 MySQL `conversations` 表加载该 chat_id 最近 30 条消息
     - ≤10 条：原文注入 `session.context["context_summary"]`
     - >10 条：调用 LLM 压缩为 ≤300 字摘要
     - 失败时 fallback 取最近 10 条原文
   - 新增 `_summarize_history(conversation_text)` 方法：调用 call_qwen 生成摘要

**数据流**：
```
message_converter: context["conversation_id"] = chat_id
→ HTTP POST /chat/ (context 在请求体中)
→ chat.py: executor.run(context=request.context)
→ executor: chat_id = context["conversation_id"]
→ _get_or_create_session(chat_id=chat_id)
→ Redis miss → 创建新 Session → _load_history_as_context()
→ MySQL SELECT → 构建对话文本 → 存入 session.context["context_summary"]
→ turn.py: 已有逻辑把 context_summary 注入 system prompt 末尾
→ LLM 获得完整上下文
```

## ai_kefu: Redis session TTL 从 30 分钟延长到 24 小时

**改动**：`config/settings.py` 中 `redis_session_ttl` 从 `1800`（30分钟）改为 `86400`（24小时）。
**理由**：闲鱼客服场景下用户可能隔较长时间回复，30分钟 TTL 太短会频繁触发新 session 创建。Redis 本身有 maxmemory-policy（LRU）兜底内存回收，不需要靠短 TTL 来管理内存。

## ai_kefu: 已有 session 也补充 context_summary + session_mapper TTL 统一

**验证结果**：之前误以为数据库时间是北京时间，实际 UTC 07:27 = 北京时间 15:27。所有 agent_turns 都是新代码运行后产生的。

- 新 session（ID 99-104，15:55~15:56，session `8fff7d2b`）**正确包含了 context_summary**：Redis miss 后 `_load_history_as_context` 从 MySQL 加载了之前 session 的对话记录，注入了 system prompt ✅

**额外修复**：
1. **`executor.py` `_get_or_create_session()`**：加载已有 session 时，如果 `messages <= 2` 且没有 `context_summary`，也从 MySQL 加载历史补充上下文
2. **`session_mapper.py` `RedisSessionMapper`**：
   - 新增 `MAPPING_TTL = 86400`（24小时）
   - `get_or_create()` 的 forward/reverse/meta mapping 都使用 `setex` 带 TTL（原来 forward/reverse 用的是 `set` 无 TTL）
   - `update_activity()` 同时续期 forward/reverse mapping 的 TTL
   - `set_manual_mode()` 的 meta TTL 也统一为 24 小时（原来 1 小时）

## ai_kefu: LLM 回复质量分析脚本 + Prompt/知识库改进分析

**新增脚本**：`scripts/analyze_llm_quality.py`
- 从 MySQL 提取对话记录 + agent_turns 详细数据
- 配对用户消息和 AI 回复，提取 tool_calls/context 注入情况
- 支持 `--auto-eval` LLM 自动评分（专业性/有用性/简洁度/准确性/语气）
- 输出终端报告 + HTML 可视化报告（保存到 `reports/` 目录）
- 用法：`python -m ai_kefu.scripts.analyze_llm_quality --limit 10 --auto-eval`

**分析发现的 Prompt / 知识库核心改进点**：
1. **回复过于冗长**：闲鱼场景下用户期望简短回复，AI 回复动辄 5-6 行（简洁度评分普遍 6/10）
2. **"238包单程顺丰"等报价消息不理解**：用户在报自己的价格/条件，AI 不理解是卖家语境
3. **反复要求收货地址**：用户多次被问同一信息，体验差
4. **[卡片消息]处理不当**：重复回复"请用文字告诉我"，没有识别系统消息类型
5. **非租赁场景处理差**：对"打气"、评价、发货通知等消息过度响应
6. **日期解析"4.27号"格式不识别**：parse_date 工具对"4.27"这种带点号的格式解析失败
7. **知识库缺少**：无砍价话术、无多台租赁优惠、无"卖家报价"识别规则
